# 个人账户与门店账户隔离设计

## 1. 目标

在保留现有门店体系和历史数据的前提下，新增面向所有用户开放的个人注册体系：

- 所有人都可以通过昵称、手机号、短信验证码和自设密码注册个人账户，无需管理员审批。
- 个人账户没有用户管理、门店资源、推广邀请等门店管理能力。
- 个人账户可申请升级为门店；管理员审批通过后创建一套全新的门店账户。
- 个人账户和门店账户允许使用相同手机号，但认证、算力、存储、图片、视频和任务数据完全隔离。
- 门店主账号和所有门店子用户必须在独立门店登录页输入门店码、账号和密码。
- Web 前端与 `miniapp-uni` 小程序执行相同的账户域隔离规则。

## 2. 保留的现有行为

- 现有 `users` 表继续作为门店账户表，保存 `MERCHANT_OWNER`、`MERCHANT_ADMIN`、`STAFF` 和 `TRIAL`；`SYSTEM_ADMIN` 也暂时保留在该表，避免无关的管理员迁移。
- 门店创建的所有子账户继续写入 `users`，通过 `merchant_id` 归属门店。
- 现有 `merchants`、门店审批、门店用户管理、门店额度池及角色规则继续使用。
- 上线前已经存在的门店账号、图片、任务、额度和日志不复制、不转移、不重新分配所有者。
- 系统管理员继续从主登录页面登录，不新增管理员专用页面。

## 3. 账户模型

### 3.1 个人账户表

新增 `personal_users`，只存个人账户：

- `id`
- `phone`：仅在 `personal_users` 内唯一，不与 `users.phone` 或 `merchants.phone` 冲突检查
- `display_name`
- `password_hash`
- `quota_balance`
- `storage_limit_bytes`、`storage_used_bytes`
- `status`
- 微信身份字段（用于小程序个人登录）
- 创建和更新时间

个人账户角色统一为 `PERSONAL`，不写入现有 `users`。

### 3.2 统一业务所有者

由于现有业务表的 `user_id` 外键只指向 `users`，新增 `account_principals` 作为业务所有者锚点：

- `id` 与对应的 `users.id` 或 `personal_users.id` 一致。
- `account_domain` 为 `SYSTEM`、`MERCHANT` 或 `PERSONAL`。
- 现有 `users` 在迁移时补写 principal；以后创建门店账号、子用户或个人用户时，都在同一事务中创建 principal。
- 图片、任务、公告已读、反馈、额度日志等业务表的所有者外键改为指向 principal，而不是只能指向 `users`。

这样账户资料仍然物理分表，业务记录又能保持数据库级引用完整性。

### 3.3 业务域字段

所有包含用户数据的业务记录增加 `account_domain`：

- 个人记录：`PERSONAL`
- 门店记录：`MERCHANT`
- 系统公共资源：`SYSTEM`

重点覆盖：图片、AI 任务、图片处理任务、工作流运行、模型调用日志、反馈、公告已读、兑换记录、系统操作日志及以后增加的视频记录。当前项目没有独立视频表；视频任务或产物必须沿用同一所有权规则。

现有普通业务记录根据当前 `user_id`、`merchant_id` 和资源作用域回填：门店业务为 `MERCHANT`，平台公共资源为 `SYSTEM`。迁移只补充所有权信息，不移动原记录。

## 4. 认证与令牌

### 4.1 令牌内容

JWT 至少包含：

- `id`
- `role`
- `accountDomain`
- 门店账户的 `merchantId`

认证中间件根据 `accountDomain` 选择 `personal_users` 或 `users` 回查，不能只根据手机号或裸 `id` 猜测账户类型。历史令牌在短期兼容窗口内按其 `users` 记录识别为 `MERCHANT` 或 `SYSTEM`；新登录签发的令牌必须显式包含账户域。

### 4.2 主登录入口

Web 使用 `#/login`，小程序使用主登录视图：

- 默认是个人登录和个人注册。
- 个人注册字段为昵称、手机号、短信验证码、自设密码。
- 注册验证码使用独立场景 `PERSONAL_REGISTER`，消费后立即创建账户。
- 个人登录支持手机号密码和手机号短信验证码。
- 页面内保留一个轻量的“系统管理员登录”切换，管理员继续使用现有管理员账号登录。
- 主登录入口不能登录门店主账号或门店子用户；检测到门店账号时提示前往门店登录。

### 4.3 门店登录入口

Web 使用 `#/merchant/login`，小程序提供独立门店登录视图。所有门店角色必须填写：

- 门店码
- 账号
- 密码

门店码使用现有六位 `merchants.merchant_code`。现有 `invite_code` 是推广来源码，不参与认证。后端先用门店码锁定 `merchant_id`，再在该门店的 `users` 中匹配账号，最后校验密码和门店状态。即使个人账户与门店账户手机号相同，门店登录也只会查询指定门店的 `users`，不会命中 `personal_users`。

旧的无门店码门店密码登录通道不再对新客户端开放，避免绕过租户定位。Web、小程序及仓库内其他登录调用必须同步切换。

## 5. 注册、额度与升级

### 5.1 个人注册

- 手机号在 `personal_users` 中不存在才允许注册。
- 同一手机号已经存在于 `users` 或 `merchants` 时，仍允许个人注册。
- 密码由用户注册时设置，沿用现有密码最小长度规则并使用 bcrypt 哈希。
- 注册成功后直接签发 `PERSONAL` 令牌，无需审批。
- 初始算力来自系统配置 `personal_registration_quota`，默认值为 `100`；按当前标准图片成本用于约十次试用，具体扣费仍服从现有任务计费规则。
- 系统管理员可在现有系统配置页调整后续新注册账户的初始值，不追溯修改已有账户。

### 5.2 申请升级门店

- 个人中心新增“升级为门店”入口，复用现有入驻表单视觉和字段。
- 请求必须携带有效 `PERSONAL` 令牌，申请记录保存 `personal_user_id` 和升级申请类型。
- 同一个个人账户同一时间只能有一个待审核升级申请。
- 管理员继续在现有申请审核页面处理，不新增管理模块。
- 审批通过后创建新的 `merchants`、`users(MERCHANT_OWNER)` 和 `MERCHANT` principal，并返回门店码、主账号和初始密码等现有审批结果。
- 个人账户继续存在；其手机号、密码、额度和所有历史业务数据均不复制到门店账户。
- 新门店从空业务数据开始，只获得审批时配置的门店额度。

## 6. 权限与数据范围

### 6.1 个人账户

个人账户可访问：

- AI 工作台
- 个人资源
- 本人历史任务和生成结果
- 本人额度明细与兑换
- 个人中心
- 问题反馈和面向所有人的公告

个人账户不可访问：

- 用户管理
- 门店资源空间
- 门店水印配置
- 推广邀请和门店报表
- 其他门店成员的图片、视频、任务或额度

### 6.2 门店账户

- 门店主账号和门店管理员继续使用门店共享额度及当前门店管理权限。
- 普通门店子用户和体验账户继续使用现有个人额度及门店内授权范围。
- 门店查询必须同时限定 `account_domain='MERCHANT'` 和当前 `merchant_id`；普通子用户仍需追加当前 `user_id` 限制。

### 6.3 个人查询

个人查询必须同时限定 `account_domain='PERSONAL'` 和当前 `user_id`，且 `merchant_id` 必须为空。手机号永远不用于业务数据授权。

## 7. 前端改动边界

- 复用现有登录卡片、表单、按钮和响应式样式，不重构整体视觉。
- 将现有 `Login.jsx` 的逻辑拆成可复用外壳、个人/管理员登录内容和门店登录内容，避免继续扩大单文件。
- 主登录页增加个人注册模式和管理员登录切换；原公开门店申请入口改为引导先注册个人账户。
- 门店登录页只增加门店码字段和入口切换链接。
- `AppShell` 为 `PERSONAL` 使用不含用户管理和推广项的导航集合。
- 小程序保持当前页面风格，新增个人注册模式、账户类型入口和门店码字段，不重做业务页面 UI。
- Web 与小程序都根据令牌返回的 `accountDomain` 决定导航和资源空间，不根据手机号或页面来源推断。

## 8. 接口边界

新增或明确以下接口职责：

- 个人注册、个人密码登录、个人短信登录
- 系统管理员登录
- 门店码密码登录
- 当前账户信息，返回 `accountDomain`
- 个人提交升级申请和读取申请状态
- 管理员审批升级申请

短信验证码场景必须区分个人注册、个人登录、密码重置和门店申请，验证码不能跨场景复用。

## 9. 错误处理与安全规则

- 门店码、账号或密码任一错误统一返回“门店码、账号或密码错误”，避免泄露门店及账号是否存在。
- 个人手机号已注册时返回明确提示，并引导登录或重置密码。
- 被禁用的个人账户、门店账户或门店均拒绝签发新令牌。
- 后端每个业务接口都以令牌账户域为准；前端隐藏菜单只用于体验，不作为权限控制。
- 所有创建账户、审批升级、扣减额度和创建业务记录的多表写入使用事务。
- 个人与门店账户可以手机号相同，但微信 `openid` 绑定也必须按账户域处理，不能自动串联两个账户。

## 10. 迁移与上线顺序

1. 创建 `account_principals` 和 `personal_users`。
2. 将现有 `users` 回填为 `SYSTEM` 或 `MERCHANT` principals。
3. 为现有业务表增加并回填 `account_domain`，保留原主键、路径和业务数据。
4. 调整相关外键和索引，使业务所有者可引用两类账户。
5. 上线支持账户域的新认证中间件和接口，同时保持旧令牌在有效期内可回查现有 `users`。
6. 同步上线 Web 与小程序的新登录入口，关闭无门店码的门店密码登录。
7. 最后开放个人注册和升级申请入口。

迁移必须可重复执行；任一步失败都不能删除或覆盖原有门店数据。

## 11. 验收标准

- 同一个手机号可以同时注册个人账户并拥有门店主账号或门店子账号。
- 个人注册立即成功，初始算力默认为 100，管理员调整配置后只影响后续注册。
- 个人账户无法访问用户管理接口和页面。
- 门店主账号及子用户缺少门店码时无法登录，错误门店码不能命中其他门店账号。
- 个人令牌无法读取同手机号门店账户的任何图片、视频、任务、资源或额度。
- 门店令牌无法读取同手机号个人账户的数据。
- 个人升级审批后可以分别登录个人入口和门店入口，两边数据保持独立。
- 上线前已有账号和历史数据仍可按原门店关系访问，记录数量和所有者不发生变化。
- Web 与小程序执行一致的账户域及门店码规则。
- 后端测试、前端测试、Web 构建、小程序合约测试和微信小程序编译全部通过。

## 12. 非目标

- 不迁移、合并或共享个人与门店历史数据。
- 不重做工作台、资源库、任务历史和用户管理页面。
- 不改变现有 AI 计费公式；100 点仅作为可配置的新用户初始额度。
- 不在本次新增管理员专用登录页面。
- 不在本次实现个人和门店账户的一键身份切换；用户需从对应入口分别登录。
