# frontend-next 工作室双端视觉一致性设计

## 目标

让 `frontend-next` 的工作室在 PC Web 与手机 Web 上复现旧 `frontend` 工作台已经确认的页面层级、视觉密度和操作路径，同时保持新前端的样式边界清晰，不继承旧前端的覆盖式 CSS 结构。

旧前端只作为视觉和交互基准，不复制其 JSX、CSS 规则或历史覆盖链。

## 范围

- 仅修改 `frontend-next` 中工作室页面及其直接依赖的局部组件、样式和测试。
- 保留现有 API、任务、上传、资源筛选、分辨率、比例和算力逻辑。
- 不修改旧 `frontend`、后端、桌面端、Android 或小程序。
- 不重新设计品牌、颜色体系、导航信息架构或业务功能。

## 布局设计

### PC Web

- 工作室主体使用全宽三栏结构：功能与资源区、主画布区、生成设置区。
- 主画布是视觉焦点；左右栏保持紧凑，并在可用高度内独立滚动。
- 去掉当前三张悬浮大卡片造成的割裂感，以清晰边界和连续工作区替代。
- 顶部状态条保留功能、当前模式、分辨率、比例和最近生成信息。
- 中等宽度优先压缩左右栏，不把右侧设置提前移动到主画布下方。

### 手机 Web

- 首屏优先显示工作室状态、主画布和上传入口。
- 功能与资源区转换为左侧抽屉，由页面内明确按钮打开和关闭。
- 生成设置转换为底部操作区或右侧抽屉；主生成按钮在单手可达区域保持可见。
- 参考图、特殊要求、分辨率和比例沿用同一份状态，不创建手机专用业务副本。
- 保留移动端底部全局导航，但避免与工作室生成操作区重叠。

## 组件边界

- `StudioPage` 负责数据、状态和工作室整体编排。
- 功能资源面板、主画布、生成设置和移动端工作室工具栏按职责拆分为局部组件。
- PC 与手机共用同一组组件和状态；响应式只改变布局、可见方式和抽屉状态。
- 不创建 `StudioPageMobile`、`StudioPageDesktop` 等重复页面。

## 样式规则

- `src/styles/tokens.css` 继续作为唯一真实颜色源。
- 工作室样式放在页面局部样式中；组件专属规则跟随组件。
- 不新增 `legacy`、`overrides`、`fixes` 等覆盖层。
- 不使用 `!important` 解决常规级联问题。
- 不复制旧前端的 `.topApp .wb*` 选择器或规则。
- 修改后删除无 DOM 对应、被新结构替代或重复声明的工作室规则。
- 每个选择器只承担一个明确职责，避免通过不断追加媒体查询覆盖同一属性。

## 交互与状态

- 功能切换、资源筛选、资源选择、原图与参考图上传、分辨率和比例选择保持可用。
- 手机抽屉打开时提供遮罩、关闭按钮和明确的可访问名称。
- 从 PC 切换到手机断点时，不丢失当前功能、资源、图片或生成设置。
- 抽屉关闭后焦点返回触发按钮；生成中和禁用状态继续可见。

## 测试策略

- 先增加失败的结构和样式边界测试，约束单一页面、抽屉入口、禁止覆盖层、禁止 `!important` 和禁止未使用的工作室选择器。
- 运行 TypeScript 检查和生产构建。
- 在浏览器中验证 1440×900、1024×768 和 390×844。
- 交互验证至少覆盖功能切换、手机功能抽屉、手机生成设置、分辨率切换和上传入口。
- 对照旧前端截图检查首屏层级、三栏比例、移动端主画布优先级、遮挡、横向溢出和底部导航冲突。

## 验收标准

1. PC 首屏与旧版同为连续三栏工作区，主画布占据主要空间。
2. 手机首屏不再被完整功能列表占满，功能和设置通过抽屉或底部操作区访问。
3. PC 与手机使用同一业务组件和状态，不存在重复页面实现。
4. 工作室样式不存在新增覆盖层、`!important`、旧 CSS 复制或无效选择器。
5. TypeScript、测试和构建通过，目标交互在三个视口无明显溢出、遮挡或滚动陷阱。

## 非目标

- 不追求旧前端历史 CSS 的逐像素复制。
- 不在本次工作中补齐资源库、历史任务或用户管理页面。
- 不改变后端接口或生成业务规则。
