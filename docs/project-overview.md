# 勋港家具 AI 修图平台项目说明

## 1. 项目定位

本项目是面向家具门店和家具商品运营场景的 AI 修图平台。核心目标是把门店日常重复的家具商品图处理流程产品化，让门店人员通过上传原图、选择功能、选择资源或填写补充要求，即可生成适合展示、上架、宣传和素材管理的家具图片。

当前项目不是单纯展示页，而是包含官网首页、登录体系、门店工作台、资源库、用户管理、任务记录、平台管理后台、AI 模型配置、公告系统、额度系统和 PWA 基础能力的一套完整业务系统。

## 3. 当前可调用的模型类型

当前后端通过 `backend/src/ai/providerService.js` 做统一模型分发。实际是否能成功调用，取决于后台模型配置中的 Provider、模型名、接口路径、API Key 和模型平台返回格式。

### 3.1 已有适配入口

| Provider | 别名 | 当前能力说明 |
| --- | --- | --- |
| `mock` | `local`, `demo` | 本地模拟模型，用于跑通业务流程 |
| `custom` | 无 | 通用 HTTP JSON 接口，适合接自定义模型服务或中转服务 |
| `zhipu` | `bigmodel`, `bigmodelcn`, `bigmodel.cn`, `zhipuai`, `cogview`, `glm` | 智谱 / CogView 类图片生成接口 |
| `gpt-image-2` | `gptimage2`, `openai`, `gptimage` | OpenAI 风格图片生成接口，包含尺寸映射和图片处理逻辑 |
| `aliyun` | `tongyi`, `wanx`, `wanxiang` | 阿里 / 通义万相入口，目前复用通用 HTTP JSON 调用 |
| `jimeng` | `volcengine`, `volc` | 即梦 / 火山入口，目前复用通用 HTTP JSON 调用 |

### 3.2 当前图片功能

这些功能都可以在模型配置中指向同一个模型，也可以分别配置不同模型：

- 材质替换：`material`
- 场景融合：`replace_bg`
- 背景净化：`remove_bg`
- 摄影增强：`enhance`
- 线稿图：`lineart`
- 多角度视图：`multiview`

### 3.3 当前视频模型

视频生成 `video_generate` 只是预留，不等于已经能生成视频。要正式调用视频模型，还需要新增视频 adapter、任务类型、结果存储、前端提交和视频预览逻辑。

## 4. HTTPS 到位后的配置项

HTTPS 证书申请完成后，需要配置到服务器和项目环境中。

### 4.1 Nginx / 网关

建议：

- 80 端口重定向到 443。
- 443 配置证书和私钥。
- 前端静态文件由 Nginx 托管。
- `/api` 代理到后端 Node 服务。
- `/files` 如果继续走后端本地文件，则代理到后端；如果使用 OSS/CDN，则图片 URL 直接走 OSS/CDN 域名。

### 4.2 后端环境变量

线上建议配置：

```env
NODE_ENV=production
APP_ENV=production
PUBLIC_BASE_URL=https://你的后端或统一域名
FRONTEND_ORIGIN=https://你的前端域名
AUTO_INIT_DB=false
RESET_LEGACY_TABLES=false
```

如果图片已经走 OSS：

```env
STORAGE_DRIVER=oss
OSS_REGION=oss-cn-hangzhou
OSS_BUCKET=your-bucket-name
OSS_ACCESS_KEY_ID=your-access-key-id
OSS_ACCESS_KEY_SECRET=your-access-key-secret
OSS_PUBLIC_BASE_URL=https://你的图片域名
```

### 4.3 前端环境变量

如果前后端同域，推荐继续使用：

```env
VITE_API_BASE_URL=/api
```

如果前后端分离域名，则配置为：

```env
VITE_API_BASE_URL=https://你的后端域名
```

Vite 的环境变量通常在构建时生效，因此修改后需要重新构建前端。

## 6. API Key 是否必须更正式加密

当前阶段如果 API Key 只保存在后端环境变量或后端数据库中，并且不会返回给前端，那么普通用户在浏览器里看不到模型 API Key。

所以内部试运行或早期上线阶段，可以先不做复杂的密钥加密系统。

但仍需注意：

- 不要把 API Key 写入前端代码。
- 不要把真实 `.env` 提交到 GitHub。
- 不要把 API Key 返回给任何前端接口。
- 服务器账号权限要收紧。
- 生产库和备份文件要限制访问。

如果未来多人运维、接入更多模型平台、产生较高调用成本，再考虑 KMS、数据库字段加密或密钥轮换。

## 7. 并发量上来后的处理方式

AI 图片生成属于慢任务，不能用普通同步接口硬扛并发。当前项目已经有异步任务结构，后续扩容建议按阶段处理。

### 7.1 第一阶段：单机可控并发

适合早期门店数量不多时：

- 限制每个模型的最大并发。
- 限制单用户同时运行任务数。
- 限制单门店同时运行任务数。
- 超时任务自动失败或回收。
- 前端保持轮询任务状态。

当前模型配置中已有 `maxConcurrency`、`timeoutMs`、`pollIntervalMs` 等字段，可继续完善使用。

### 7.2 第二阶段：任务队列

当并发增长后，需要从“请求直接执行模型调用”升级为“请求只创建任务，worker 消费任务”。

建议结构：

- API 服务只负责鉴权、扣费预占、创建任务。
- Worker 服务负责调用模型。
- Redis / BullMQ / RabbitMQ / 数据库队列表负责排队。
- 任务状态写回 MySQL。
- 前端继续轮询或使用 WebSocket/SSE。

这样可以避免用户请求长时间占用 Node 进程。

### 7.3 第三阶段：多实例扩容

当任务更多时：

- 后端 API 多实例部署。
- Worker 多实例部署。
- 按模型供应商拆分 worker。
- 按任务类型设置不同队列。
- 热门模型独立限流。
- 对失败任务做重试和熔断。

### 7.4 第四阶段：成本和稳定性控制

需要加入：

- 模型平台限流保护。
- 任务失败自动退还或部分退还算力。
- 单用户频率限制。
- 单门店每日额度限制。
- 模型错误告警。
- OSS 上传失败告警。
- 慢任务统计。
